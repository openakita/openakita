## 二、消息完整生命周期

### 2.1 消息入口与路由

```
 用户消息 (Telegram/Feishu/CLI/...)
    │
    ▼
 ┌─────────────────────────────────────────────────────┐
 │  Gateway._on_message()                              │
 │  ├─ /restart → 系统重启                              │
 │  ├─ 会话正在处理中?                                   │
 │  │   ├─ 停止/stop → cancel_event → 中断当前任务       │
 │  │   ├─ 跳过/skip → skip_event → 跳过当前工具         │
 │  │   └─ 普通消息 → 加入中断队列(稍后注入对话)           │
 │  └─ 否 → 加入 _message_queue                        │
 └──────────────────────┬──────────────────────────────┘
                        │
                        ▼
 ┌─────────────────────────────────────────────────────┐
 │  Gateway._process_loop()                            │
 │  ├─ 系统命令拦截 (/model, /thinking, /new, /reset)   │
 │  │   └─ 直接回复,不经过 Agent                        │
 │  ├─ 媒体预处理 (图片下载, 语音→文字, base64编码)       │
 │  └─ 调用 Agent._call_agent(session, message)        │
 └──────────────────────┬──────────────────────────────┘
                        │
                        ▼
 ┌─────────────────────────────────────────────────────┐
 │  Agent.chat_with_session()                          │
 │  ├─ _prepare_session_context() → 构建消息列表         │
 │  ├─ _chat_with_tools_and_context()                  │
 │  │   ├─ 构建系统提示词 (PromptAssembler)             │
 │  │   └─ 委托给 ReasoningEngine.run()                │
 │  └─ _finalize_session() → 存记忆, 记录指标            │
 └─────────────────────────────────────────────────────┘
```

```

 ║                    │                                 ║

 ║          ▼                   ▼                      ║
 ║   TOOL_CALLS              FINAL_ANSWER              ║
 ║          │                   │                      ║
 ║          ▼                   ▼                      ║

 ║  │ ACT (执行):    │   │ 验证是否完成   │              ║
 ║  │ ToolExecutor   │   │ → 是: 返回    │              ║
 ║  │ .execute_batch()│   │ → 否: 回到推理│              ║

 ║  └───────┬───────────┘                              ║
 ║          │                                          ║
 ║          └──→ 回到 REASON ──→ 循环                   ║
```

### 2.3 消息数据结构

```python

    "role": "user" | "assistant",

        {"type": "text", "text": "..."},
        {"type": "tool_use", "id": "...", "name": "...", "input": {...}},
        {"type": "tool_result", "tool_use_id": "...", "content": "..."},
        {"type": "image", ...},
        {"type": "thinking", "thinking": "..."}  # Claude 3.7+ 思考链
```

### 2.4 工具执行细节

```
 ToolExecutor.execute_batch(tool_calls)
    │
    ├─ 并行决策:
    │   ├─ 默认串行 (允许中断检查)
    │   ├─ 可选并行 (tool_max_parallel > 1)
    │   └─ 浏览器/桌面/MCP: 互斥锁保护
    │
    ├─ 单工具执行:
    │   handler_registry.execute_by_tool(name, input)
    │   → 路由到对应 handler (filesystem/browser/shell/MCP等)
    │
    ├─ 结果截断保护:
    │   result > 16,000 字符 → 保存完整输出到溢出文件 + 截断
    │
    └─ 返回: [{tool_use_id, content, is_error}, ...]
```

### 2.5 响应发送与中断处理

```
 Agent返回响应文本
    │
    ▼
 ┌─────────────────────────────────────────────────────┐
 │  Gateway._send_response()                           │
 │  ├─ 按平台限制分割:                                   │
 │  │   ├─ Telegram: 4,096 字符/消息                    │
 │  │   ├─ 微信: 2,048 字符/消息                        │
 │  │   └─ 其他: 2,000 字符/消息                        │
 │  ├─ 每段重试3次发送                                   │
 │  └─ 段间速率限制                                      │
 └──────────────────────┬──────────────────────────────┘
                        │
                        ▼
 ┌─────────────────────────────────────────────────────┐
 │  处理中断队列中的待处理消息                            │
 │  _process_pending_interrupts()                      │
 │  ├─ 取出中断消息                                     │
 │  ├─ 加入会话历史                                     │
 │  ├─ 再次调用 Agent                                   │
 │  └─ 发送响应 → 循环                                  │
 └─────────────────────────────────────────────────────┘
```

### 2.6 循环终止条件

| 条件 | 结果 |
|------|------|
| 最终回答被接受 | 正常返回响应 |
| 达到最大迭代次数 | 返回 "task timeout" |
| 用户取消 | 返回告别消息 |
| 资源预算超限 | 返回预算警告 |
| 验证失败 | 重新进入循环 |

## 三、系统提示词组装

系统提示词是 LLM 看到的"上帝视角"，决定了 Agent 的行为模式。

### 3.1 组成结构

```
 System Prompt 组成结构
 ═══════════════════════════════════════════════
 ┌──────────────────────────────────────────┐
 │  1. 身份信息 (AGENT.md + SOUL.md)        │  ← "我是谁"
 │     角色定义、性格特征、行为准则           │
 ├──────────────────────────────────────────┤
 │  2. 系统环境信息                          │  ← "在哪里运行"
 │     OS、工作目录、临时目录                 │
 ├──────────────────────────────────────────┤
 │  3. 环境快照 (每轮动态更新)               │  ← "当前状态"
 │     ├─ 目录结构                           │
 │     ├─ 计划状态 (PlanHandler)             │
 │     ├─ 预算信息 (Token/迭代/时间)         │
 │     ├─ 最近3条错误                        │
 │     └─ 草稿板/工作记忆                    │
 ├──────────────────────────────────────────┤
 │  4. 技能目录 (70+ skills)                │  ← "会什么"
 │     渐进式展示: 名称+描述                  │
 ├──────────────────────────────────────────┤
 │  5. MCP 服务目录                          │  ← "外部能力"
 ├──────────────────────────────────────────┤
 │  6. 工具列表 (89+ tools)                 │  ← "有什么工具"
 │     按类别: 文件系统/技能/记忆/浏览器等    │
 ├──────────────────────────────────────────┤
 │  7. ★ 记忆注入区 ★                       │  ← "记得什么"
 │     ├─ 记忆系统说明                       │
 │     ├─ 草稿板 (当前任务)                  │
 │     ├─ 用户钉选规则 (PINNED, 不受预算裁剪)│
 │     ├─ 核心记忆 (MEMORY.md, ≤1500字)     │
 │     └─ 历史经验提示 (高价值记忆)           │
 ├──────────────────────────────────────────┤
 │  8. 用户信息 (USER.md)                    │  ← "用户是谁"
 ├──────────────────────────────────────────┤
 │  9. 工具使用指南 + 核心原则               │  ← "怎么行动"
 │     三层工具系统、永不放弃原则             │
 └──────────────────────────────────────────┘
```

### 3.2 Token 预算分配

```
 ├─ 身份区: ~60% 的预算
 ├─ 目录区: 33%(工具) + 55%(技能) + 10%(MCP)
 ├─ 记忆区: 弹性分配(使用剩余空间)
 └─ 每个区域都有 apply_budget() 截断保护
```

### 3.3 两个版本

- **v1 (prompt_assembler.py)**: 非编译版，作为兼容回退
- **v2 (prompt/builder.py)**: 编译版，检查缓存是否过期并按需重编译，节省约55% token消耗

### 3.4 工具使用指南要点

三层工具系统:
1. **系统工具** — 渐进式展示 (列表 → 详情 → 执行)
2. **技能** — 渐进式展示，支持加载/重载
3. **MCP服务** — 全量暴露

核心四法则:
1. 任务请求**必须**使用工具
2. 没有工具? 创建一个
3. 自己解决问题
4. **永不放弃**

## 四、三层记忆系统

### 4.1 架构全景

```
 ┌─────────────────────────────────────────────────────────────────┐
 │                    三层记忆架构                                  │
 │                                                                 │
 │  Layer 1: 草稿板 (Scratchpad)                                   │
 │  ┌────────────────────────────────────────────────┐             │
 │  │  跨会话持久工作记忆                              │             │
 │  │  内容: 当前任务、活跃项目、未解决问题、下一步      │             │
 │  │  存储: SQLite scratchpad 表                     │             │
 │  │  注入: 始终注入系统提示词顶部                     │             │
 │  └────────────────────────────────────────────────┘             │
 │                                                                 │
 │  Layer 2: 核心记忆 (MEMORY.md + Pinned Rules)                   │
 │  ┌────────────────────────────────────────────────┐             │
 │  │  人工策划的用户事实和永久规则                     │             │
 │  │  格式: Markdown (identity/MEMORY.md)            │             │
 │  │  区段: 用户偏好 / 重要规则 / 关键事实             │             │
 │  │  上限: 1500字符 (规则段优先保留)                  │             │
 │  │  注入: 始终注入系统提示词                         │             │
 │  │  特殊: PINNED规则有单独区域,不受预算裁剪影响      │             │
 │  └────────────────────────────────────────────────┘             │
 │                                                                 │
 │  Layer 3: 动态记忆 (Semantic + Episode)                         │
 │  ┌────────────────────────────────────────────────┐             │
 │  │  AI自动提取的知识,语义索引检索                    │             │
 │  │                                                 │             │
 │  │  SemanticMemory (实体-属性结构):                  │             │
 │  │  ├─ FACT: 事实                                   │             │
 │  │  ├─ PREFERENCE: 偏好                             │             │
 │  │  ├─ SKILL: 技能经验                              │             │
 │  │  ├─ ERROR: 错误教训                              │             │
 │  │  ├─ RULE: 规则                                   │             │
 │  │  ├─ PERSONA_TRAIT: 人格特征                      │             │
 │  │  └─ EXPERIENCE: 经历                             │             │
 │  │                                                  │             │
 │  │  Episode (完整交互故事):                           │             │
 │  │  └─ 目标、结果、操作节点、使用工具、相关实体        │             │
 │  │                                                  │             │
 │  │  存储: SQLite + FTS5全文索引 (+可选ChromaDB向量)  │             │
 │  │  注入: 高价值部分主动注入, 其余按需 search_memory │             │
 │  │  生命周期:                                       │             │
 │  │    TRANSIENT(1天) → SHORT_TERM(3天)              │             │
 │  │    → LONG_TERM(30天) → PERMANENT(永久)           │             │
 │  └────────────────────────────────────────────────┘             │
 └─────────────────────────────────────────────────────────────────┘
```

### 4.2 记忆创建流程

```
 对话进行中...
    │
    ├──→ 话题切换时: extract_on_topic_change()
    │
    └──→ 会话结束时: end_session()
              │
              ▼
 ┌─────────────────────────────────────────────┐
 │  Extractor.extract_from_conversation()       │
 │  ├─ 输入: 最近30轮对话                       │
 │  ├─ LLM分析: 提取用户身份/偏好/规则信息      │
 │  ├─ 不提取: 一次性任务请求、具体文件细节      │
 │  └─ 输出: [{type, subject, predicate,        │
 │            content, importance, duration}]    │
 └──────────────────┬──────────────────────────┘
                    │
                    ▼
 ┌─────────────────────────────────────────────┐
 │  多层去重 (Multi-Layer Dedup)                │
 │  ├─ 精确匹配: subject+predicate 完全一致    │
 │  │   → 演进已有记忆 (提升confidence)         │
 │  ├─ 语义相似: bigram overlap > 0.3           │
 │  │   → LLM确认是否重复                       │
 │  └─ 无匹配 → 创建新记忆                      │
 └──────────────────┬──────────────────────────┘
                    │
                    ▼
 ┌─────────────────────────────────────────────┐
 │  持久化 (双写)                               │
 │  ├─ SQLite memories 表 (权威源)              │
 │  ├─ FTS5 全文索引 (自动触发器同步)            │
 │  ├─ SearchBackend (FTS5/ChromaDB/EmbeddingAPI)│
 │  └─ memories.json (v1兼容双写)               │
 └─────────────────────────────────────────────┘
```

### 4.3 三种提取模式

| 模式 | 触发时机 | 是否需要LLM | 提取内容 |
|------|---------|------------|---------|
| extract_from_conversation | 话题切换/会话结束 | 是 | PREFERENCE, RULE, FACT, PERSONA_TRAIT |
| extract_experience | 会话结束 | 是 | EXPERIENCE, SKILL, ERROR (可复用模式) |
| extract_quick_facts | 上下文压缩前 | 否(规则匹配) | 匹配"每次..."、"不要..."、"必须..."模式 |

### 4.4 重要性与优先级映射

```python
if importance >= 0.85 or mem_type == RULE:
    priority = PERMANENT       # 永不过期
elif importance >= 0.6:
    priority = LONG_TERM       # 30天
else:
    priority = SHORT_TERM      # 3天
```

### 4.5 引用评分机制

```
 retrieve() 返回记忆给 Agent
    ↓
 record_cited_memories(memories)  — 缓存在会话中
    ↓
 提取时传入 cited_memories 给 LLM
    ↓
 LLM 评估每条记忆的有用性: [{memory_id, useful}]
    ↓
 有用的记忆 → bump_access() 增加访问计数
    ↓
 访问计数影响后续检索排序权重
```

### 4.6 记忆检索与注入

#### 方式1: 主动注入 (每次构建系统提示词)

```
 build_system_prompt()
    └─ _build_memory_section()
        ├─ Layer 0: 记忆系统说明文本 ("你有一个三层分层记忆网络...")
        ├─ Layer 1: scratchpad.get() → 当前任务
        ├─ Layer 1.5: query(type=RULE, pinned=true) → 钉选规则
        │             (不受记忆预算裁剪)
        ├─ Layer 2: read(MEMORY.md) → truncate_memory_md(≤600字)
        └─ Layer 3: query(type∈{EXPERIENCE,SKILL,ERROR},
                          importance≥0.6,
                          sort_by=access_count*importance,
                          limit=10)
                    → 历史经验提示
```

#### 方式2: 按需检索 (Agent主动调用 search_memory 工具)

```
 Agent → tool_call: search_memory(query="...")
    └─ retrieval_engine.retrieve(query, recent_messages, max_tokens=700)
        ├─ 查询分解 (LLM提取关键词 + 意图分类)
        │   意图: search_memory | search_file | general
        │
        ├─ 四路搜索:
        │   ├─ 语义搜索 (FTS5/向量匹配)
        │   ├─ Episode搜索 (实体名匹配)
        │   ├─ 近期高价值记忆 (3天内, importance≥0.6)
        │   └─ 附件搜索 (文件名/描述/标签)
        │
        ├─ 合并去重 (同一记忆保留最高相关性)
        │
        ├─ 重排序:
        │   score = 0.40 × relevance
        │         + 0.20 × recency (指数衰减)
        │         + 0.20 × importance
        │         + 0.20 × access_freq (log尺度, 上限1.0)
        │
        └─ Token预算裁剪 (累积直到≤700 tokens)
```

## 五、上下文压缩机制 (ContextManager)

当对话越来越长时如何管理上下文窗口——这是整个系统最关键的组件之一。

### 5.1 上限计算

```
 context_window = 模型上下文窗口 (默认200K, 最低8192)
 output_reserve = min(max_tokens, context_window / 3)
 max_available  = (context_window - output_reserve) × 0.95  ← 5%安全余量
 hard_limit     = max_available - system_prompt_tokens - tools_tokens - 500
 soft_limit     = hard_limit × 0.85                         ← 触发压缩阈值
```

### 5.2 Token 估算

```python
# 中英文感知的 token 估算
中文: ~1.5 字符/token
英文: ~4 字符/token
每条消息: +10 token 结构开销
多模态: 文本+图片+工具分别计算
最小: 1 token
```

### 5.3 消息分组策略

```
 分组规则 (保证工具调用/结果不被拆开):
 ┌─────────────────────────────────────────┐
 │  assistant 消息 (含 tool_use)           │
 │  + user 消息 (含对应 tool_result)       │ → 绑定为一组
 │  + user 消息 (含更多 tool_result)       │
 └─────────────────────────────────────────┘
 ┌─────────────────────────────────────────┐
 │  其他消息                                │ → 独立分组
 └─────────────────────────────────────────┘
```

### 5.4 六级压缩决策树

```
 当前消息总token > soft_limit ?
    │
    ├─ 否 → 不压缩, 直接使用
    │
    └─ 是 ↓

 ┌────────────────────────────────────────────────────────┐
 │  Stage 1: 通知记忆系统                                  │
 │  memory_manager.on_context_compressing(messages)        │
 │  → extract_quick_facts() (规则式, 无LLM, 极快)          │
 │  → 保存关键事实到记忆库                                  │
 └──────────────────────┬─────────────────────────────────┘
                        ▼
 ┌────────────────────────────────────────────────────────┐
 │  Stage 2: 大型工具结果压缩                               │
 │  tool_result > 5,000 tokens → 独立LLM压缩               │
 │  (保留 OVERFLOW_MARKER 标记的不压缩)                     │
 └──────────────────────┬─────────────────────────────────┘
                        ▼
 ┌────────────────────────────────────────────────────────┐
 │  Stage 3: 上下文边界感知压缩                             │
 │  检测 [上下文边界] 标记 (话题切换点)                      │
 │  → 边界前的消息用 18% 压缩率激进压缩                      │
 └──────────────────────┬─────────────────────────────────┘
                        ▼
 ┌────────────────────────────────────────────────────────┐
 │  Stage 4: 分组压缩 (核心策略)                            │
 │  保留最近 8 组不压缩                                     │
 │  早期组 → LLM摘要压缩到 15%                              │
 │  摘要注入为 [之前的对话摘要] 用户消息                     │
 └──────────────────────┬─────────────────────────────────┘
                        ▼
 ┌────────────────────────────────────────────────────────┐
 │  Stage 5: 递归压缩 (仍超限时)                            │
 │  保留组从 8 减到 4, 再次压缩                              │
 └──────────────────────┬─────────────────────────────────┘
                        ▼
 ┌────────────────────────────────────────────────────────┐
 │  Stage 6: 硬截断 (最终安全阀)                            │
 │  丢弃最早的消息直到满足 hard_limit                        │
 │  丢弃的消息 → 加入记忆提取队列                            │
 │  插入提示: [context_note: 早期对话已自动整理]              │
 └──────────────────────┬─────────────────────────────────┘
                        ▼
 ┌────────────────────────────────────────────────────────┐
 │  Post: 压缩后方向注入 (确定性, 无LLM)                    │
 │  rewrite_after_compression():                           │
 │  ├─ 任务描述预览                                        │
 │  ├─ 当前计划状态                                        │
 │  ├─ 已执行工具列表                                      │
 │  └─ 草稿板/工作记忆                                     │
 │  → 追加到最后一条用户消息 (防止"失忆")                    │
 └─────────────────────────────────────────────────────────┘
```

### 5.5 摘要压缩的内容保留策略

`_summarize_messages_chunked()` 将消息按 30K token 分块，每块要求 LLM 保留:

- 对话背景
- 用户需求
- 已完成步骤
- 当前进度
- **待解决问题** (原文保留)
- **关键配置/数值** (端口、路径、版本号原文保留)
- 下一步计划
- **用户行为规则** (精确保留)

## 八、设计原则总结

1. **消息不断累积** — ReAct 循环中 user/assistant/tool_result 持续增长
2. **记忆是消息的"蒸馏"** — 在压缩/话题切换/会话结束时，从消息中提取高价值信息存入记忆
3. **上下文管理是"遗忘"的艺术** — 6 级渐进式压缩，从智能摘要到硬截断，始终保证最近对话完整
4. **记忆反哺上下文** — 通过系统提示词注入 + 按需搜索两种方式，让"遗忘"的信息可以被重新召回
5. **工具交互完整性** — 分组策略保证 tool_use/tool_result 永不被拆开
6. **SQLite 是唯一真相源** — JSON、向量库、FTS5 都是二级同步副本，优雅降级
7. **确定性优先** — 压缩后方向注入不使用 LLM，避免引入不确定性
8. **中文感知** — Token 估算区分中英文字符，压缩提示词使用中文
