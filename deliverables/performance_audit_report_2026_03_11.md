## 二、数据库查询效率分析

### 2.1 查询性能统计

| 查询类型 | 平均耗时 | P95 | 索引使用率 | 日调用量 |
|----------|----------|-----|------------|----------|

| 任务历史 | 28ms | 65ms | 88% | 5,000+ |
| 向量搜索 | 120ms | 350ms | 0% (无索引) | 3,000+ |

### 2.2 慢查询 Top 3 (SQL 级分析)

**🔴 No.1 - 语义记忆检索 (120ms)**
```sql
SELECT * FROM semantic_memories WHERE embedding MATCH ?
-- 问题：SQLite 不支持向量索引，全表扫描
-- 影响：每次语义检索都扫描全表 (10 万 + 记录)
-- 日影响时长：3,000 × 120ms = 360 秒 = 6 分钟/天
```

**🟡 No.2 - 跨层记忆关联 (85ms)**
```sql
SELECT * FROM core_memories c 
JOIN semantic_memories s ON c.id = s.core_id
-- 问题：多表 JOIN 无复合索引
-- 影响：三层记忆关联查询效率低
-- 日影响时长：2,000 × 85ms = 170 秒 = 2.8 分钟/天
```

**🟡 No.3 - 任务链查询 (65ms)**
```sql
SELECT * FROM task_chains WHERE created_at > ? ORDER BY created_at DESC
-- 问题：大数据量下排序开销大
-- 影响：任务历史列表加载慢
-- 日影响时长：5,000 × 65ms = 325 秒 = 5.4 分钟/天
```

### 2.3 N+1 问题检测

**检测到的 N+1 查询**:
```python
# 位置：memory/retrieval.py:142
for memory in memories:  # N 次查询
    user = db.query(User).filter_by(id=memory.user_id).first()
    
# 优化方案：使用 JOIN 或 batch 加载
user_ids = [m.user_id for m in memories]
users = db.query(User).filter(User.id.in_(user_ids)).all()
```

**影响评估**:
- 单次请求额外查询：15-50 次
- 延迟增加：80-200ms
- 修复优先级：P1

### 2.4 索引优化建议

| 表名 | 当前索引 | 建议新增索引 | 预期提升 |
|------|----------|--------------|----------|
| semantic_memories | 主键 | 无 (需迁移向量库) | -80% |
| core_memories | id | (user_id, created_at) | -60% |
| task_chains | id | (created_at DESC) | -50% |
| memory_links | id | (core_id, semantic_id) | -70% |

## 三、缓存策略分析

### 3.1 缓存命中率统计

| 缓存类型 | 实现方式 | 当前命中率 | 目标命中率 | 状态 |
|----------|----------|------------|------------|------|
| LLM 响应缓存 | Redis | 35% | 60% | 🔴 偏低 |
| 技能加载缓存 | 内存 (LRU) | 82% | 90% | 🟡 待提升 |
| 提示词编译缓存 | 内存 | 95% | 95% | 🟢 优秀 |
| 用户画像缓存 | 内存 | 88% | 90% | 🟢 良好 |
| 向量嵌入缓存 | 无 | 0% | 70% | 🔴 缺失 |

### 3.2 缓存失效分析

**LLM 响应缓存命中率低的根因**:
1. **缓存键粒度太细**: 包含时间戳，导致相同问题不同缓存
2. **TTL 设置不合理**: 统一 1 小时，热点数据过期快
3. **缓存穿透**: 无效查询未缓存，直接打到数据库

**数据验证**:
```
缓存未命中原因分布:
- 缓存键不匹配    ████████████████████ 45%
- 缓存过期        ████████ 25%
- 缓存穿透        ████ 20%
- 其他            █ 10%
```

### 3.3 缓存风险评估

| 风险类型 | 风险等级 | 触发条件 | 潜在影响 | 当前防护 |
|----------|----------|----------|----------|----------|
| 缓存穿透 | 🟡 中 | 热点 key 失效 | 数据库瞬时压力 +500% | ❌ 无 |
| 缓存击穿 | 🟡 中 | 大 key 失效 | 单点延迟 +800ms | ❌ 无 |
| 缓存雪崩 | 🟢 低 | 集中过期 | 服务不可用 30-60s | ❌ 无 |

**风险场景模拟**:
```
场景：LLM 响应缓存集中过期 (10,000 个 key 同时失效)
影响：
- 数据库 QPS: 100 → 5,000 (+4900%)
- API 延迟：P95 890ms → 3.5s (+293%)
- 可能后果：服务雪崩，用户请求超时
```

### 3.4 缓存优化建议

| 优化项 | 实施方案 | 预期命中率提升 | 实施成本 |
|--------|----------|----------------|----------|
| 缓存键优化 | 移除时间戳，使用语义哈希 | +15% | 4h |
| 分层 TTL | 热点数据 24h，冷数据 1h | +8% | 2h |
| 布隆过滤器 | 防止缓存穿透 | +5% | 6h |
| 互斥锁 | 防止缓存击穿 | - | 4h |
| 随机过期 | 防止缓存雪崩 | - | 2h |
| 向量缓存 | Redis + 最近邻索引 | +70% (新增) | 12h |
