## 快速入门

### 第一步：创建组织

1. 打开组织编排页面
2. 点击左侧面板的「从模板创建」选择预置模板，或手动创建空组织
3. 在画布上添加节点（员工），连线定义汇报关系

### 第二步：选择运营模式

在右侧组织设置面板顶部切换运营模式：
- **指令模式**：组织等待你下达任务指令
- **自主经营模式**：组织根据配置的核心业务自主运营

### 第三步：创建项目与任务

1. 点击底部面板的「项目」标签
2. 创建新项目（如「官网改版」）
3. 在项目中添加任务，设置标题、描述、优先级

### 第四步：启动组织

点击顶部工具栏的「启动」按钮。

### 第五步：派发任务或等待自主执行

- **指令模式**：在项目面板中点击任务卡片上的「派发」按钮，任务会下发给组织 CEO 节点
- **自主经营**：CEO 节点会自动根据核心业务制定计划并执行

### 第六步：查看进度

- **节点悬浮窗**：鼠标悬浮在节点上查看当前任务、计划进度、委派情况
- **任务详情**：点击项目面板中的任务卡片查看完整详情
- **节点属性 → 任务标签**：选中节点后在右侧面板切换到「任务」标签
- **运营看板**：切换到看板视图查看整体运营数据

## 核心概念

### 组织 (Organization)

一个完整的多 Agent 团队，包含节点、连线、配置。每个组织有独立的运营模式、项目和任务系统。

### 节点 (Node)

组织中的一个 Agent 成员。每个节点有：
- **角色**：职位名称和目标
- **部门**：所属部门
- **层级 (level)**：0 = CEO/顶层，1 = 副总裁，2 = 总监，3+ = 员工
- **状态**：idle（空闲）、busy（工作中）、waiting（等待）、error（错误）、frozen（冻结）
- **工具**：配置的 MCP 服务器、技能和组织内工具
- **头像**：SVG 预置头像或自定义上传

### 连线 (Edge)

定义节点间的汇报关系。类型包括：
- `reporting`：汇报关系（默认）
- `collaboration`：协作关系
- `communication`：沟通关系

### 项目 (Project)

任务的组织单元，分为：
- **永久项目**：持续性业务（如"日常运营"）
- **临时项目**：限时任务（如"Q2 营销活动"）

### 任务 (Task)

项目中的具体工作项。每个任务有：
- **task_id**：全局唯一标识（如 `task_abc123`），用于追踪和查询
- **状态**：todo → in_progress → delivered → accepted / rejected
- **进度**：0-100% 的完成百分比
- **负责人**：被分配的节点
- **委派者**：分配任务的节点
- **子任务**：支持无限层级的子任务拆解
- **Plan**：执行计划步骤
- **执行日志**：所有状态变更的时间线记录

### chain_id（内部概念）

消息路由层面的内部标识，用于关联同一任务链的消息。用户无需关心，系统自动管理。

## 两种运营模式

### 指令模式 (Command Mode)

适用场景：用户给组织下达明确的工作指令。

工作流程：
1. 用户创建项目和任务
2. 用户点击「派发」将任务下发给组织
3. CEO 节点接收任务，根据需要委派给下级
4. 各节点执行并汇报进度
5. 用户查看进度、验收结果

特点：
- 不自动启动业务
- 心跳仅做健康检查
- 看门狗恢复卡死节点后不自动重新激活

### 自主经营模式 (Autonomous Mode)

适用场景：让组织自主规划、执行、复盘业务。

工作流程：
1. 用户配置组织的「核心业务」描述
2. 启动组织
3. CEO 自动根据核心业务制定计划
4. 组织自主执行、委派、协作
5. 心跳触发经营复盘，调整策略

特点：
- 自动启动（auto_kickoff）
- 空闲检测（idle_probe）主动催促节点寻找工作
- 看门狗恢复后自动重新激活

## 任务管理

### 创建任务

**方式一：项目面板手动创建**
1. 在底部项目面板中选择或创建项目
2. 添加任务，填写标题和描述

**方式二：Agent 自动创建**
当 Agent 使用 `org_delegate_task` 委派任务时，系统自动创建对应的子任务。

**方式三：Agent 主动创建**
Agent 可使用 `org_create_project_task` 工具在项目中创建任务。

### 派发任务

在项目面板中，待办状态的任务卡片上有「派发」按钮：
- 点击后任务状态变为 `in_progress`
- 任务内容作为指令发送给组织 CEO
- chain_id 自动关联，后续所有执行记录都与此任务绑定

### 任务状态流转

```
todo → in_progress → delivered → accepted
                                → rejected → in_progress (重新执行)
```

### 多级子任务

当 CEO 委派任务给 CTO，系统自动创建子任务：
```
task_001: 整体规划 (CEO, depth=0)
├── task_002: 技术方案 (CTO, depth=1)
│   ├── task_005: 前端架构 (架构师, depth=2)
│   └── task_006: 后端API (架构师, depth=2)
├── task_003: 产品需求 (CPO, depth=1)
└── task_004: 市场计划 (CMO, depth=1)
```

- 无层级限制，完全由组织层级自然形成
- 子任务完成时，自动递归重算所有祖先任务的进度

## 多级子任务与 Plan

### Plan 工具

所有组织节点默认可用以下 Plan 工具：
- `create_todo`：创建执行计划（多个步骤）
- `update_todo_step`：更新某一步骤状态
- `get_todo_status`：查看计划当前进度
- `complete_todo`：标记计划完成

### Plan 与任务的联动

Plan 工具的执行会自动桥接到项目任务系统：
- `create_todo` → 将计划步骤写入任务的 `plan_steps`
- `update_todo_step` → 同步更新进度，自动计算 `progress_pct`
- `complete_todo` → 记录完成事件

### 进度计算规则

- **叶子任务**：进度由节点通过 `org_report_progress` 或 Plan 步骤完成比例自动计算
- **非叶子任务**：`progress_pct = avg(所有子任务的 progress_pct)`
- **已验收任务**：视为 100%
- **已打回任务**：视为 0%

## 进度追踪与可观测性

### 节点悬浮窗

鼠标悬浮在节点上，可看到：
- 角色、部门、层级、当前状态
- 当前任务标题和 Plan 进度条
- 已委派子任务完成情况
- 运行时长、外部工具、最近活动时间
- 看门狗状态（如最近是否被恢复）

### 节点属性 → 任务标签

选中节点后，在右侧面板切换到「任务」标签：
- 当前任务 + 执行计划
- 分配给此节点的所有任务列表
- 此节点委派的任务及进度

### 任务详情面板

点击项目面板中的任务卡片，打开详情面板：
- 完整信息：ID、标题、状态、负责人
- 面包屑导航：显示祖先任务链，可点击跳转
- 子任务树：可展开折叠，每个显示状态和进度
- Plan 步骤列表：✓ 已完成 / → 进行中 / ○ 待办
- 执行时间线：所有事件按时间排列

### 运营看板

切换到看板视图可看到大屏风格的运营数据概览。

### Agent 查询工具

Agent 自身也可使用工具查询进度：
- `org_list_my_tasks`：查看自己的任务
- `org_list_delegated_tasks`：查看委派的任务
- `org_get_task_progress`：查看任务详情（含子任务树）
- `org_report_progress`：主动汇报进度

## 看门狗守护机制

看门狗是一个运行时级别的后台守护协程，监控组织健康状态。

### 检测规则

| 场景 | 条件 | 处理 |
|------|------|------|
| BUSY 卡死 | 节点 BUSY 超过阈值（默认 30 分钟）且无任何活动 | 取消任务、重置状态、通知用户 |
| ERROR 不恢复 | 节点持续 ERROR | 清理缓存、重置为 IDLE |
| 全组织沉默 | 所有节点 IDLE 超过阈值（仅自主模式） | 重新激活 CEO |

### 模式差异

- **自主经营**：看门狗更积极，恢复后自动重新激活
- **指令模式**：看门狗更保守，恢复后仅通知用户

### 配置项

在组织设置中可调整：
- `watchdog_enabled`：是否启用（默认开启）
- `watchdog_interval_s`：检查频率（默认 30 秒）
- `watchdog_stuck_threshold_s`：卡死阈值（默认 1800 秒）
- `watchdog_silence_threshold_s`：沉默阈值（默认 1800 秒）

### 与旧超时机制的区别

旧系统使用 `asyncio.wait_for` 强制掐断任务（900 秒超时），复杂任务经常被误杀。

新系统完全取消硬超时，改为看门狗基于"活跃度"判断：
- 只要节点在持续产生活动（工具调用、消息、事件），就视为正常
- 只有长时间无任何活动才判定为卡死

## 常见问题与排查

### Q: 组织启动后节点一直 IDLE，不执行任何工作

**指令模式**：这是正常的。需要你在项目面板中创建任务并点击「派发」。

**自主经营模式**：检查是否已配置「核心业务」。如果核心业务为空，CEO 不知道该做什么。

### Q: 看门狗通知"节点因长时间无活动被恢复"

通常表示 LLM 调用出现了异常（连接断开、API 超时等）。看门狗已自动恢复节点。可在活动流中查看详细事件。

### Q: 子任务进度没有更新到父任务

检查子任务是否设置了 `parent_task_id`。自动创建的子任务（通过 `org_delegate_task`）会正确设置。手动创建时需要指定父任务 ID。

### Q: 节点无法委派任务

确保节点有下级节点（画布上有连线）。系统会自动为有下级的节点授予委派权限。

### Q: 如何查看节点的执行计划

1. 鼠标悬浮节点查看 Plan 进度条
2. 选中节点 → 右侧面板 → 「任务」标签
3. API：`GET /api/orgs/{org_id}/nodes/{node_id}/active-plan`

### Q: 如何查看完整的任务执行历史

点击项目面板中的任务卡片，在详情面板底部可以看到完整的执行时间线，包括所有状态变更、委派、进度更新等事件。
