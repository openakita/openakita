# OpenAkita v1.27.13 — Release Notes

> Patch release on top of v1.27.12. User-visible focus is the factory
> security default; everything else is the source-of-truth refactor that
> backs it and migration guidance for upgraders.

## TL;DR

- Fresh installs now boot into the **trust** security profile by default
  (was `protect`).  This makes the recommended low-interrupt experience
  the out-of-the-box experience, in line with the multi-agent flow that
  v1.27.12 users reported as "too noisy compared to 小秋".
- All matrix safety nets stay on: DESTRUCTIVE and UNKNOWN tools still
  hit a confirm dialog; `safety_immune` paths still block; `death_switch`
  still triggers; the sandbox still classifies high-risk shell commands.
- Existing `POLICIES.yaml` files are honoured untouched. You only see a
  behaviour change if you do not have a YAML, or your v1 YAML has no
  explicit `confirmation.mode` (see "BC notes" below).
- Schema defaults, the `_apply_security_profile_defaults` bundle, and
  the setup-center loading placeholder now all read from a single
  source of truth in `policy_v2/defaults.py`. No more "UI says trust
  while engine runs protect" drift.

## What changed for users

### Factory security profile = `trust`

Open SecurityView on a fresh install and the "信任方案 / Trust" card is
highlighted by default. Previously a fresh install silently sat in
`protect` even though the in-repo `identity/POLICIES.yaml` already
shipped as trust — the inconsistency came from `protect` being baked
into the Pydantic schema defaults that fresh installs fall back to.

Trust is **not** "yolo bare-metal". The decision matrix routes
`DESTRUCTIVE` (anything `delete_file` / shell `rm -rf /` shaped) and
`UNKNOWN` (any tool the classifier has never seen) through `CONFIRM`
under every mode except `dont_ask`. The protective layers that do not
depend on `confirmation.mode` also stay armed:

| Layer | Default in v1.27.13 |
|---|---|
| `confirmation.mode` | `trust` (matrix: most classes ALLOW, DESTRUCTIVE / UNKNOWN CONFIRM) |
| `profile.current` | `"trust"` (UI label only — engine does not read this for decisions) |
| `sandbox.enabled` | `True` (high-risk shell commands still run via `CommandSandbox.check_command`) |
| `shell_risk.enabled` | `True` |
| `death_switch.enabled` | `True` |
| `checkpoint.enabled` | `True` |
| `safety_immune` paths | unchanged (identity / audit / sessions / credentials etc.) |

There is an intentional asymmetry between the schema default and the
"信任方案" bundle that the SecurityView card writes: clicking the card
also turns `sandbox.enabled` off (that's the bundle's UX promise), while
the schema default keeps it on as a belt-and-suspenders fallback. This
is now documented in the `PolicyConfigV2` docstring.

### SecurityView first-paint matches the engine

The setup-center loading placeholder used to render as "保护方案" /
`default` for a frame before the API fetch returned the real values.
Both placeholders now match the new factory default, so first-paint and
fetched state agree.

## BC notes for upgraders

### You have a `POLICIES.yaml` written by v1.27.x setup-center

You are unaffected. Setup-center writes `profile.current` and
`confirmation.mode` explicitly, so the schema-default change cannot
reach you. Your existing profile and mode survive the upgrade
verbatim.

### You have a v1 (pre-policy-v2) YAML with an explicit mode

Also unaffected. `LEGACY_MODE_ALIASES` still translates
`yolo / smart / cautious` to `trust / default / strict`.

### You have a v1 YAML with **no** `confirmation` block

This is the one path where v1.27.13 silently changes runtime behaviour.
Before v1.27.13 the missing `confirmation.mode` field defaulted to
`default` (= v1 `smart` = protect-ish). After v1.27.13 it defaults to
`trust`.

To detect this, the loader now emits an INFO log line when it sees a
v1 schema YAML without an explicit `confirmation.mode`. Sample:

```
[PolicyV2] /path/to/POLICIES.yaml is v1 schema without explicit
security.confirmation.mode; v1.27.13+ uses factory default 'trust'
(低打扰档，DESTRUCTIVE/UNKNOWN 仍 CONFIRM). To keep the v1.27.x
'protect/default' behaviour add `security.confirmation.mode: default`
to your YAML.
```

If you want the old behaviour back, add to your YAML:

```yaml
security:
  confirmation:
    mode: default
```

…or run the migration via setup-center's "保护方案 / Protect" card
once and it'll write a full v2 bundle for you.

### Env override behaviour

`OPENAKITA_AUTO_CONFIRM=true` still forces `confirmation.mode = trust`
unconditionally. `OPENAKITA_AUTO_CONFIRM=false` still forces it to
`default`. Env overrides remain orthogonal to schema defaults; the
behaviour of these flags is unchanged.

## What changed for developers

### Single source of truth: `policy_v2/defaults.py`

Before v1.27.13 the "what does each profile bundle contain" question
had answers in three independent places:

1. `policy_v2/schema.py` field defaults (`SecurityProfileConfig.current`,
   `ConfirmationConfig.mode`).
2. `api/routes/config.py::_apply_security_profile_defaults` — the
   handler that fires when a user clicks one of the four cards.
3. `apps/setup-center/src/views/SecurityView.tsx` — the React loading
   placeholder.

These were not kept in sync (that's exactly how v1.27.12 ended up
shipping with `profile=protect` schema defaults but UI assumptions
written against `trust`). v1.27.13 consolidates this:

```python
# src/openakita/core/policy_v2/defaults.py
FACTORY_DEFAULT_PROFILE = "trust"

PROFILE_BUNDLES = {
    "trust":   {"enabled": True,  "confirmation": {"mode": "trust"},   ...},
    "protect": {"enabled": True,  "confirmation": {"mode": "default"}, ...},
    "strict":  {"enabled": True,  "confirmation": {"mode": "strict"},  ...},
    "off":     {"enabled": False, "confirmation": {"mode": "trust"},   ...},
}

def factory_default_confirmation_mode() -> ConfirmationMode: ...
def factory_default_profile_current() -> str: ...
def profile_bundle(profile: str) -> dict: ...
```

The schema fields now consume `factory_default_*()`;
`_apply_security_profile_defaults` consumes `profile_bundle()`. Tests
in `tests/unit/test_policy_v2_profile_bundles.py` pin the cross-
source agreement so a future bundle tweak can't drift back.

If you ever need to change the factory default profile, the place to
edit is `FACTORY_DEFAULT_PROFILE`. The schema, the routing layer, the
setup-center initial state, and the upgrade-INFO log message will all
update from that one constant.

### `_mode_from_security` fallback semantics

The routing-layer helper that derives a v1 product-mode label from a
v2 `security` dict used to return `"smart"` (= protect) when the dict
existed but had no `confirmation` block. It now returns `"yolo"`
(= trust), aligned with the schema default. This affects only the
`/api/config/permission-mode` GET response for YAML files with a
non-empty security section but no confirmation field — a very rare
shape in practice, but worth calling out for anyone parsing those
fields downstream.

### Public surface additions to `policy_v2.defaults`

| Name | Kind | Purpose |
|---|---|---|
| `FACTORY_DEFAULT_PROFILE` | `str` constant | "Which profile is the factory default" — single edit point |
| `PROFILE_BUNDLES` | `dict[str, dict]` | Read-only registry of baked profile bundles |
| `profile_bundle(name)` | `(str) -> dict` | Deep-copied bundle by name (raises `KeyError` for `custom` / unknown) |
| `factory_default_confirmation_mode()` | `() -> ConfirmationMode` | For schema field default |
| `factory_default_profile_current()` | `() -> str` | For schema field default |

These are exported through `policy_v2/__init__.py` to keep imports
short for downstream code.

## Tests

`tests/unit/test_policy_v2_profile_bundles.py` is the new contract
file. Existing migration / loader / API tests were updated to assert
the v1.27.13 defaults. Full suite (31 policy_v2 / security-config test
files) was green at +14 contract tests on top of v1.27.12.
