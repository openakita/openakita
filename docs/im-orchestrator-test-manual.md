# 聊天 / IM 接管「组织编排指挥台」用户测试手册

本手册写给**普通用户**：你打开设置中心，在聊天框或者用手机/桌面 IM 客户端跟 Bot 对话，想验证「组织（Org）指挥台」这项功能是否好用。**整个手册不要求你写代码，也不要求你调任何 API**——所有操作要么是在桌面前端点鼠标，要么是给 Bot 发文字消息。

---

## 一、你需要先理解的几件事

**什么是「组织（Org）」？**  组织是把多个 AI 角色（例如「文案策划」「美术指导」「项目经理」）按上下级关系拼成的一棵树。给组织下一条指令，组织内的多个角色会按职责分工、协作产出结果。这跟跟单个 AI 助手对话（比如直接问"小秋"）不一样——单个助手就一个人在干，组织是一队人在干。

**什么是「指挥台」？**  指挥台就是你给组织下指令、看它当前在干什么、必要时叫停它的那一套界面/命令。本手册里"指挥台"出现在两个地方：

1. **桌面前端**（设置中心里的「组织编排」面板和「聊天」页右上角的「组织模式」开关）——有按钮可点、有列表可看。
2. **IM 客户端**（Telegram、飞书、钉钉、企微、QQ、OneBot 等）——只能用文字命令，例如 `/org` 开头。

**关于「组织名字」与「组织 ID」**：每个组织都有两个标识：

- **名字**：你创建组织时取的中文名，比如「内容工作室」。**全局唯一**——重名会被系统直接拒绝。
- **组织 ID**：系统自动生成的一串字母数字（类似 `org_3f7a9b2e1c08`），后台用来精确定位。

由于名字唯一，**你在 IM 端可以直接用名字下达指令**，不需要记一串 hex ID。例如：

```
/org bind 内容工作室
@组织 帮我写一段产品文案
```

或者一行式：

```
/org 内容工作室 帮我写一段产品文案
```

名字匹配是**大小写不敏感、首尾空白不敏感**的（`Content Studio`、`  content STUDIO  `、`Content Studio` 都视为同一个组织）。如果名字里恰好包含空格、不便在 IM 里输入，建议你在桌面前端把组织改个不带空格的短名（例如把「Content Studio」改成「content-studio」），改名时同样受唯一性保护。

如果一些遗留数据里仍存在历史重名，IM 里输入这个名字时 Bot 会主动列出所有候选并给出每个组织的 ID 与创建时间，让你用更精确的名字或直接用 ID 再试一次。

**IM 里的"指挥台"控制指令**：在 IM 任何时刻——**包括 `@组织 <任务>` 正在跑、Bot 还没回话的时候**——你都可以发送下面三条命令立刻得到响应：

- `/org running` 或 `/组织 在跑`：查看当前会话在跑哪条组织命令、跑到什么阶段、忙碌节点是哪些。
- `/org cancel` 或 `/组织 取消`：立刻取消当前在跑的组织命令；组织本身保持运行，可以马上发下一条。
- `/org last` 或 `/组织 上次`：重看上一条已完成命令的结果。

## 二、测试前你要先准备好的东西

**1. 启动 OpenAkita 服务**

在你日常启动 OpenAkita 的方式里启动它（桌面 App 双击，或者命令行里 `openakita serve`）。等设置中心能在浏览器里打开就行。

**2. 创建并启动一个组织**

打开设置中心，进入「组织编排」页面（左侧导航里能找到）。

- 点「新建组织」，从模板里挑一个（比如「内容工作室」），保存。
- 创建好后，组织列表里会出现这一行，状态可能是「未启动」或类似字样。点这一行进入详情，再点「启动」按钮，等状态变成「运行中」（`RUNNING`）。

**3. 记下组织名字**（IM 测试要用）

由于新版组织名字全局唯一，**你只需要记住名字**就能在 IM 端使用，不再需要去抄一串 `org_xxx` ID。建议你把名字取得短一点、不带空格，例如「内容工作室」「研发组」等。

如果你确实想看 ID（例如要去 REST API 里精确指定），打开桌面前端的组织详情页，组织 ID 应当显眼地展示在页眉或基本信息卡片上。找不到的话可以临时按 F12 → Network 看 `/api/orgs/<ID>` 请求里的那串字符——但**正常用户测试不需要这一步**。

**4. 配置至少一个 IM Bot**（如果你要测 IM 部分）

打开设置中心的「消息通道」页，启用一个 Bot（推荐 Telegram，找 @BotFather 申请一个 token 粘贴进来就行）。保存后稍等几秒，确认页面提示 Bot 已上线。

**5. 准备一段统一的「任务文本」**

为了多个用例之间能横向对比，建议你后面所有用例都用同一段任务文字。这里给你一段：

```
请为「家用智能投影仪」准备一份发布方案：
1) 一段约 200 字的产品宣传文案；
2) 3 条小红书风格的推广文案；
3) 一份简短的传播节奏建议（按周拆分）。
```

后续用例里凡是说"发送任务文本"或"粘贴任务文本"，都是指这一段。

## 三、在桌面聊天端的测试

桌面聊天端有两种使用方式，覆盖了大多数用户的日常体验。

### 3.1 用「组织编排面板」直接对组织下指令

这是最直观的一种方式：你点开组织、在它的对话面板里给它派活。

操作步骤：

第一步，回到设置中心的「组织编排」页，点开刚才创建并启动的那个组织。

第二步，找到这个组织的「对话/编排面板」（OrgChatPanel），它通常就在组织详情页里，有一个输入框和发送按钮。

第三步，把第二章准备好的「任务文本」粘贴到输入框里，发送。

预期看到的现象：

- 发送后立刻出现一条系统消息，类似「**命令已派发，命令 ID：xxxxxxxx**」（这串 ID 是这一次指令的编号，不是组织 ID，别搞混）。
- 紧接着会陆续出现「**组织进度**」类提示，例如「内容策划节点 任务完成」「美术指导节点 任务完成」等，每一条对应组织内某个节点干完了一步。
- 等所有节点都干完，会出现一条**最终结果**消息，里面是聚合好的、可以直接交付的完整方案。

判定通过：你能看到三类信息（派发回执 / 至少一条进度提示 / 最终聚合结果），并且最终结果包含了你任务里要求的三个部分（产品文案 + 3 条小红书 + 传播节奏）。如果中间某个节点报错，但最终聚合结果里如实说明了"某节点未完成原因"，也算通过——只要不是完全卡死或前端报错。

### 3.2 在指令执行过程中按「取消」

这一条紧接着 3.1：你再发送一次任务文本（或者发送一条更长更难的任务，例如把字数要求改到 800 字），在它**还在跑**的时候，去面板上找「取消」按钮（一般在那条命令消息旁边，或者输入框附近，文案大概是"取消"/"中止"/"停止当前指令"）。

点一下取消按钮，期望：

- 几秒钟内出现一条提示「**已取消**」或类似措辞。
- 组织状态仍然是「运行中」，**不会**变成「已停止」——这是设计如此，组织本身没关闭，你可以立刻再下一条新指令。
- 立刻再发一条短一点的任务（比如「写一句 30 字的口号」），它应当能正常被接收并完成。

判定通过：「取消」按了能停、不报错、组织还在线、能继续接活。

### 3.3 同一时刻不能下两条指令（同根冲突保护）

继续在 3.1 的面板里：发送一条任务文本后，**别等它结束**，立刻再发一条不同的任务（比如「随便写一段欢迎语」）。

期望：第二条会被前端立刻挡回来，提示类似「**组织已有命令正在执行，请稍后再试或显式取消/替换**」。

如果前端提供了"替换/抢占"的选项（按钮文字可能是「强制替换」「抢占运行」），点它确认；预期看到第一条指令被自动取消、第二条立刻开始运行。

判定通过：第二条被正确拦截、有清晰提示、用户能选择"等"或"抢"。

### 3.4 用「聊天页 + 组织模式开关」下指令

这种方式更接近"日常聊天"：你不去组织编排页，就在普通聊天里通过开关切换。

操作步骤：

第一步，回到设置中心的「聊天」页面。

第二步，找到聊天界面右上角（或顶部某处）的「**组织模式**」开关，把它打开；下拉框会让你选哪个组织，选刚才那个。

第三步，在普通聊天输入框里发送任务文本。

预期：跟 3.1 几乎一样——会先有"命令已派发"提示，然后是若干进度，最后聚合结果替换在 AI 回复气泡里。区别仅在于"载体"不同：3.1 是组织专属面板，3.4 是普通聊天框披着"组织外套"。

判定通过：组织模式打开后能正常下发；关闭组织模式（把开关切回去）再发一句普通问候（例如「你好」），应当回到普通 AI 助手的回答风格（一个人在跟你聊，不再有"进度"和"派发"概念）。

### 3.5 续接上一次结果再追加任务

接着 3.4：上一条任务跑完后，新开一句：

```
基于上面那份方案，再补充一段约 100 字的「目标用户画像」。
```

如果界面上有「**接续上一条**」「**基于上次结果继续**」之类的选项/复选框，勾上再发。

预期：组织能"看到"上一条的产物，补充的内容和上文衔接得上（用了同一类产品定位、口吻一致），而不是从零另起一段。

判定通过：补充内容显然引用或衔接了上一条产物（例如延用了"家用智能投影仪"这个产品名、口吻一致）。

### 3.6 查组织在干什么、有没有卡住

在桌面前端，组织编排页/详情页通常有：

- **组织状态**显示（运行中 / 暂停 / 未启动 / 已归档）；
- **节点状态**展示（每个角色当前在忙/在等/已完成）；
- 也许还有一个**事件流/日志**面板。

操作：在 3.1 那条指令执行的同时，去看这些状态显示。

期望：组织状态是「运行中」；某些节点会被标为「正在工作」并显示在干什么任务；当一个节点完成、下一个节点接手时，状态会更新；指令结束后所有节点回到空闲。

判定通过：能从界面上**实时看到**组织内部的执行情况，而不是只有一个黑盒"正在处理"提示。

## 四、在 IM 里的测试

IM 端没有按钮，只能发文字命令。新版命令**优先按名字识别**，名字不行才需要用 ID。

下面 IM 用例统一以 Telegram 私聊为例。飞书、钉钉、企微、QQ、OneBot 操作完全相同，文字一样发。

### 4.1 列出可用组织（先看看你能操作哪些组织）

给你的 Bot 发：

```
/org list
```

中文等价：

```
/组织 列表
```

期望：Bot 列出当前所有未归档的组织，每行类似 `• 内容工作室  [running]  ID: org_xxxxxxxx`。

判定通过：列表里能看到第二章你创建的那个组织、有正确的运行状态。

### 4.2 把组织"绑定"到当前 IM 会话

为了不每次都打全名，先做一次绑定。给 Bot 发：

```
/org bind 内容工作室
```

中文等价：

```
/组织 绑定 内容工作室
```

期望 Bot 回复：「**已绑定组织：「内容工作室」(ID: org_xxxxxxxx)**」——同时把可读名字和真实 ID 都显示给你。

然后再发：

```
/org status
```

或中文：

```
/组织 状态
```

期望回复：「**当前绑定组织：「内容工作室」(ID: org_xxxxxxxx)**」。

判定通过：两条回复内容正确、显示的名字与 ID 一致。

如果输入了一个**不存在的名字**（比如 `/org bind 不存在的组织`），Bot 应回复：「**未找到组织「不存在的组织」。请用 `/org list` 或在桌面端查看可用组织名。**」

如果输入了一个**带歧义的名字**（仅当系统里仍存在历史重名时才可能发生），Bot 应该回复一份候选列表，每条带 ID 和创建时间，让你重新用更精确的方式（带后缀的名字或直接用 ID）指定。

如果你要解绑（一般测完不解绑也行），发：

```
/org unbind
```

回复应当是「**已取消当前 IM 会话的组织绑定。**」

### 4.3 用绑定好的组织下指令（推荐方式）

绑定之后，给 Bot 发：

```
@组织 这里粘贴第二章的任务文本（多行也可以）
```

或英文：

```
@org Paste the task text from chapter 2 here.
```

预期看到的现象（**私聊**）：

- Bot 立刻先回复一条：「**已向组织 org_xxxxxxxx 下发指令，命令 ID：xxxxxxxx**」。这是回执，让你知道指令收到了。
- 接下来 Bot 会陆续推送 1 条或多条「**组织进度：xxx 任务完成：xxx**」之类的摘要消息，每条对应组织内某一步进展。
- 最后 Bot 推送一条**完整的最终结果**，里面包含你任务要求的三个部分。

预期看到的现象（**群聊**，如果你把 Bot 拉进了一个群里 @它发命令）：

- **不会**先回那条"已下发"的回执（避免吵到群里）。
- 中间的进度摘要**会被抑制**，不会刷屏。
- 只在最后推送一条完整结果。

判定通过：私聊三类消息都有、最终结果完整；群聊只有一条最终结果且无中间噪声。如果你同时在桌面 3.1 那个面板上观察，应该能看到同一条命令在两边都有体现（IM 这边只是按摘要级别显示而已）。

### 4.4 不用绑定，直接一句话下达（完整形式）

如果你不想 bind，也可以一条命令搞定：

```
/org 内容工作室 这里粘贴任务文本
```

中文写法：

```
/组织 内容工作室 这里粘贴任务文本
```

期望：跟 4.3 完全一样。

注意点：`/org` 后面**第一个空格之前**会被当成"组织名或 ID"，**第一个空格之后**全部当成任务正文。这意味着**组织名最好不要包含空格**（如果包含空格的名字，第一个空格会被误当成分隔符）。建议你在第二章给组织取名时就避免空格——例如用 `内容工作室` 而不是 `内容 工作室`。

如果你已经有空格名的组织且暂不想改名，**用先 `/org bind <名字>` 再 `@组织 <任务>` 的两步方式**：bind 命令会把整段名字（不分空格）作为参数读取，所以可以正确解析。

### 4.5 故意制造同根冲突看错误提示

4.2 / 4.3 那条命令**还在跑**的时候，再向同一个 Bot 发一条不同的任务：

```
@组织 帮我想一句 30 字以内的传播口号。
```

期望：Bot 几乎立刻回复一条：「**组织命令提交失败：组织已有命令正在执行，请稍后再试或显式取消/替换。**」

判定通过：第二条没有"插队"开始执行；第一条原本的等待不受影响、最终仍然能正常返回结果。

### 4.6 故意发错误格式看 Bot 怎么处理

请逐一测试下面这些"错误输入"，记录 Bot 的实际反应：

第一种，**只发 `/org` 什么都不带**：当前实现下 Bot 不会有任何专门提示，这条消息会被当成普通聊天交给主 Agent 处理。**预期 Bot 当作一句"你好"或者一句不知所云的输入处理**——这一点其实不太友好，可以记到第六章建议清单里。

第二种，**`/org` 后面只带名字不带任务正文**（例如 `/org 内容工作室`）：和上面一种类似，会被当普通聊天处理。

第三种，**用一个根本不存在的名字**：例如 `/org 不存在的组织 写一段话`。预期：Bot 回复「**未找到组织「不存在的组织」。请用 `/org list` 或在桌面端查看可用组织名。**」

第四种，**`/org bind` 不带参数**：预期 Bot 回复「**用法：/org bind <组织名 或 组织ID>**」。

第五种，**没绑定就发 `@组织 写点东西`**：当前实现下不会触发组织命令通道，消息会被当成普通聊天处理。这点也不太理想，应当至少提示一下"请先 /org bind"。同样列到第六章。

第六种，**在组织指令还在跑的时候试图发"停止"或"`/stop`"**：普通的"停止/`/stop`/kill"会被网关识别为**单 Agent 任务**的中断词，对正在跑的"组织命令"**无效**。要立刻干预组织命令，请用下一节（4.7）介绍的 `/org cancel`。

### 4.7 在 IM 里实时干预正在跑的组织命令（新版三连）

这一节是验证刚刚落地的"指挥台 IM 控制面"——三条**绕过等待循环的 fast-path 指令**：`/org cancel`、`/org running`、`/org last`。它们的关键特性是**即使另一条 `/org <名字> <任务>` 还在跑、Bot 看起来"没回话"，这三条仍然能立刻得到响应**。

操作步骤如下，请按顺序做：

第一步，在 IM 里向 Bot 发送一条**比较慢的组织任务**（确保它运行至少 30 秒以上，方便观察）：

```
/org bind 内容工作室
@组织 请写一份完整的产品发布方案，包含一段约 800 字的宣传文案、5 条小红书种草文案、1 份海报创意脚本、1 份传播节奏表。请尽量详细。
```

Bot 收到后会立刻回执：「**已向组织「内容工作室」下发指令，命令 ID：`cmd_xxxxxxxx`** … 查看进度：`/org running` · 立即取消：`/org cancel` · 重看上次结果：`/org last`」

第二步，**任务还在跑的时候**，给 Bot 发：

```
/org running
```

或中文：

```
/组织 在跑
```

期望 Bot 立刻（不需要等任务结束！）回复一段卡片，类似：

```
📍 当前正在跑：「内容工作室」(ID: org_xxxxxxxx)
  • 命令 ID：`cmd_xxxxxxxx`
  • 任务：请写一份完整的产品发布方案，包含一段约 800 字的宣传文案...
  • 阶段：dispatching
  • 已运行：18 秒
  • 忙碌节点：node_writer, node_designer
如要叫停，发送 `/org cancel`。
```

如果状态拿不到（极少见，比如服务暂时抖动），Bot 仍然能根据 session 记录的快照给你看到「组织名字 + 命令 ID + 任务摘要」，不会崩。

判定通过：在 `@组织 ...` 还在跑的同时，`/org running` 能**立刻**响应（小于 1 秒），且内容里至少包含组织名 / 命令 ID / 任务摘要三项。

第三步，**仍在第一步任务运行期间**，给 Bot 发：

```
/org cancel
```

或中文：

```
/组织 取消
```

期望 Bot 立刻回复：「**✅ 已发起取消，命令 ID：`cmd_xxxxxxxx`。组织会在执行完当前最小步骤后停止；最终结果（含「cancelled_by_user」）将通过此前那条派发回执的等待循环发回。**」

紧接着，**几秒内**你之前那条 `@组织 ...` 的等待应当也结束——Bot 会发回一条最终消息（内容里通常会标注被用户取消、结果可能是半成品）。

判定通过：
1. `/org cancel` 立刻有回执（不需要等几十秒）；
2. 原本那条 `@组织 ...` 的等待循环也随之结束（不再无限阻塞）；
3. 组织本身保持「运行中」状态——你可以立刻再发下一条新任务。

第四步，重看上一条命令的结果：

```
/org last
```

或中文：

```
/组织 上次
```

期望 Bot 回复一段类似：

```
📃 上次组织命令：「内容工作室」(ID: org_xxxxxxxx)
  • 任务：请写一份完整的产品发布方案...

结果：
（这里是组织上次完整的产出文本，可能是被你取消后的半成品）
```

判定通过：能拿到上一条命令的"最终结果文本"，文本不会无故被截断到 0；session 重启后再发 `/org last` 仍然能拿到（因为存在 session metadata，跨重启可恢复）。

第五步，边界场景：

- **没有"在跑"的命令时发 `/org running`**：期望 Bot 回复「当前没有正在跑的组织命令。用 `/org bind <组织名>` 绑定后，再 `@组织 <任务>` 派发。」
- **没有"上次"记录时发 `/org last`**：期望 Bot 回复「本会话还没有任何已完成的组织命令记录。」
- **`/org cancel` 在没在跑的会话里**：期望 Bot 回复「当前没有正在跑的组织命令可以取消。」
- **服务暂未就绪时发 `/org cancel`**：期望 Bot 回复「组织命令服务尚未初始化，请稍后再试。」（这条很难手动复现，知道行为即可）

### 4.8 验证名字唯一性约束（验证新版改进已落地）

这一节验证"创建同名组织"会被系统拒绝。请按下面步骤来：

第一步，回到桌面前端的「组织编排」页，点「新建组织」，名字填**与第二章那个组织完全相同**的名字（比如又输入「内容工作室」）。

第二步，点保存。

期望：前端弹出错误提示，大意是：「**已存在同名组织「内容工作室」，请改个名字。为了在聊天和 IM 中能用名字直接调用组织，组织名字必须全局唯一。**」前端不会创建出第二个同名组织。

第三步，把名字改成不同的（比如「内容工作室 2」），再保存。预期：能正常创建。

第四步，试试**大小写差异**：如果你已有 `Content Studio`，再创建 `content studio`（全小写）应当也被拒绝；类似 `  Content Studio  `（带空白）也会被识别为重名。

第五步（可选）：试试**改名撞别人**。在「内容工作室 2」详情里把名字改成「内容工作室」，期望同样被拒绝。

判定通过：上述四种重名场景前端都给出清晰提示、不创建/不保存；只有用一个真正不冲突的名字才能成功。

## 五、桌面端补充测试（`/agent` 真切换 & 组织 ID 一键复制）

### 5.1 `/agent` 不带参数：列出可用 Agent 并标出当前

打开设置中心 → 聊天页，确保聊天框为空。在输入框里**只**敲 `/agent`（不带任何参数）并按回车。

预期：

- 不会发出聊天请求，也不会向 Bot/Agent 发送"请切换到 Agent xxx"那种自然语言。
- 聊天历史里多一条**系统提示**，列出当前实例里**所有 Agent**（id + 图标 + 名字 + 描述），其中**当前正在使用的那个**末尾带 `⬅️ 当前` 标记。
- 末尾给出用法提示「切换：`/agent <agent_id>`」。

如果你没看到 `⬅️ 当前` 标记，先确认你的 `selectedAgent` 是否在 Agent 列表里（很久未刷新的 web 会话偶发不同步，刷新一次页面应恢复）。

### 5.2 `/agent` 带 id：真切换并下发到下一条消息

继续在 5.1 的同一聊天会话里，从 5.1 列出的清单里挑一个**和当前不一样**的 agent_id（例如 `office-doc`），发送：

```
/agent office-doc
```

预期：

- 聊天历史里多一条系统提示「✅ 已切换到 📄 **办公文档助手** (`office-doc`)」（图标和中文名取决于 Agent profile 配置）。
- 同样**不会**触发任何聊天请求。
- 顶部「当前 Agent」下拉如果之前显示的是别的角色，现在会变成新选择的那个。
- 紧接着你发送一条普通用户消息（比如「介绍一下你自己」），后端日志里能看到 `agent_profile_id` 已经是 `office-doc`，并且 `_apply_agent_profile` 把它写入了 `session.context`。后续即使你再发消息不带参数，会话仍会持续使用这个新 profile，直到你下次显式切换或新开会话。

### 5.3 `/agent` 用名字片段 / 模糊匹配

发送：

```
/agent 文档
```

如果你的实例里只有一个 Agent 名字包含「文档」，预期与 5.2 一致——直接命中并切换。

如果有多个 Agent 同时包含「文档」，预期是**系统提示列出所有候选**让你用更精确的 id 再发一次，而**不会**误切换到第一个。

### 5.4 `/agent` 已经是当前 / 找不到时的友好提示

发送两条用例：

```
/agent <当前 Agent 的 id>
/agent zzzzzz_definitely_not_exists
```

第一条预期：「ℹ️ 当前已是 X」，不会重复切换、不会改历史。
第二条预期：「❌ 未找到 Agent `zzzzzz_definitely_not_exists`。…发送 `/agent` 不带参数可以查看所有 Agent。」

### 5.5 桌面 `/agents` 列表与 `/agent` 是否一致

发送 `/agents`，预期和 5.1 中**不带参数的 `/agent`** 完全等价（同样列出全部 Agent + ⬅️ 当前 标记）。差别只在底部的提示词措辞。

### 5.6 顶栏组织 ID 胶囊 + 一键复制

进入设置中心 → 组织编排，随便选一个组织。预期顶部栏组织名右侧多了一颗**虚线胶囊**「ID `org_xxxxxxxx`」+ 一个小复制图标。

- **桌面端**胶囊里显示完整 `org_xxxxxx` ID。
- **移动端**会自动缩成前 6 位 + `…`，但 tooltip 与复制到剪贴板的内容仍是**完整 ID**。
- 点击这颗胶囊，预期看到底部出现成功 toast「已复制组织 ID 到剪贴板」，去任意输入框（IM 也行）粘贴，得到的就是完整的 `org_xxxxxxxxx`。

### 5.7 左侧组织列表卡片的 mini ID + 复制

打开「组织列表」抽屉（顶栏左上角的菜单按钮），看左侧每张组织卡片副标题。预期每张卡片右下都有一颗小 chip「`org_xxxxxx…`」+ 复制图标。

- 点 chip：复制完整 ID，弹出与 5.6 一样的 toast。**不会**误触发"切换到这个组织"——chip 内部已经 stopPropagation。
- 切换组织：点卡片的**任何其他空白处**（不在 chip 上）才会真切换。

### 5.8 指挥台头部的 ID 胶囊

回到 5.6 的组织详情页，点击「组织指挥台」抽屉（或任何一处打开 `OrgChatPanel` 的入口），预期面板顶部"组织指挥中心 · 组织名"的下方多了一行同款 ID 胶囊。

- 点击胶囊：sonner 全局 toast 弹出「Organization ID copied to clipboard」（英文/中文取决于当前 UI 语言）。
- 这个胶囊不影响"清空历史""关闭抽屉"按钮的点击。

### 5.9 复制能力跨平台

如果你能切换运行环境：

- **Tauri 桌面端**：复制走 `@tauri-apps/plugin-clipboard-manager`，应当成功。
- **浏览器/Web**：在 HTTPS 或 `localhost` 下走 `navigator.clipboard`，应当成功；在非安全上下文下会回退到 `execCommand("copy")`，仍应成功并提示「已复制」。
- 极端情况下三种方式全失败，预期看到红色 toast「复制失败，请手动选中再复制」。

## 五附、CLI 命令行用户怎么办

如果你是 CLI 用户，老实说**当前版本的 CLI 里没有"组织指挥台"入口**。`openakita`（交互模式）和 `openakita run "..."`（单次模式）都是直接跟单个 AI 助手对话，不会把指令派给某个组织。

短期解决办法：

- 用桌面前端的方式（第三章），它就在浏览器里、不要求装额外东西。
- 或者把 IM Bot 配置好，跟自己的 Bot 私聊（第四章）。

长期建议：CLI 加一组 `openakita org cmd <名字或ID> "<任务>"`、`openakita org status <名字或ID>` 之类的子命令。这条也列在第六章。

## 六、本次测试要把这些建议反馈给团队

测试过程里我们会反复撞到下面这些事，请你把它们填到测试报告里：

**第 1 项（本轮已落地）**，IM 端已支持三条**绕过等待循环**的 fast-path 控制指令：`/org cancel`（立即取消当前会话正在跑的组织命令）、`/org running`（查看当前会话在跑的组织命令、附实时阶段/忙碌节点/已运行时长）、`/org last`（重新看上一条已完成命令的结果，跨进程重启可恢复）。所有英文指令都有中文别名（`/组织 取消`、`/组织 在跑`、`/组织 上次`）。**测试方法见 4.7**。

**第 2 项（本轮已落地）**，IM 现在已支持按**组织名字**直接调用。所有入口（`/org bind <名字>`、`/org <名字> <任务>`、`@组织 <任务>`、`/org list`、`/org status`）都先按名字解析、找不到再按 ID 兜底，名字匹配大小写/空白不敏感。如果由于历史数据存在重名，Bot 会主动列出候选让你消歧。**测试方法见 4.1–4.3、4.6**。

**第 3 项（本轮已落地）**，创建/改名组织时已强制名字全局唯一。重名会被后端直接 409 拒绝，前端展示统一的中文提示。`OrgManager.create / update / duplicate / create_from_template` 全路径覆盖；从模板创建时若默认名已被占用，会自动在末尾加 `(2)/(3)` 后缀保证不冲突。**测试方法见 4.8**。

**第 4 项（本轮已落地）**，IM 等待循环不再"独占会话"。新增的 `/org cancel` 三连指令在 `_on_message` 阶段 fast-path 处理、**不进 `_message_queue` 也不走 per-session 中断队列**，因此即便一条 `@组织 <任务>` 还在 `await queue.get()` 阻塞，用户也能立即用控制指令查询/取消。**测试方法见 4.7 第二、三步**。

**第 5 项（本轮已落地）**，桌面聊天页的 `/agent` 斜杠命令现在是**真切换**——直接更新当前会话的 `selectedAgent`，下一条消息会随 `agent_profile_id` 一起发给后端，由 `_apply_agent_profile` 写入 `session.context`，与 IM 的 `/切换` 语义对齐。不带参数发 `/agent`（或 `/agents`）会列出全部可用 Agent，并用 ⬅️ 标记当前正在使用的那一个；带参数 `/agent <id-或-名字片段>` 先尝试 id 精确命中，再做模糊匹配，匹配到多于一个时列出候选要求精确化，匹配不到时给出错误提示。**测试方法见 5.x（桌面侧用例）**。

**第 6 项（本轮已落地）**，桌面前端的组织列表/详情/指挥台**都已显眼地展示组织 ID**并提供一键复制按钮：
- 顶部栏组织名右侧多了一颗虚线胶囊「ID `org_xxxxxx`」，点击复制；移动端胶囊会自动缩成前 6 位 + `…`，但完整 ID 在 tooltip 与剪贴板中都是完整的。
- 左侧组织列表每张卡片副标题里加了「`org_xxxxxx…`」迷你 chip + 复制图标，点击直接复制而不会误触切换组织。
- 抽屉式「组织指挥台」（`OrgChatPanel`）头部也加了同样的 ID chip。
- 三处复制成功都会弹出 toast 反馈（顶栏/列表用本页面已有的底部 toast，指挥台用 sonner 全局 toast）。i18n 已加 `org.editor.copyOrgId / orgIdCopied / orgIdCopyFailed / orgIdMissing / copyOrgIdHint` 与 `org.chat.copyOrgId / orgIdCopied / orgIdCopyFailed`，中英文都补齐。**测试方法见 5.x（桌面侧用例）**。
