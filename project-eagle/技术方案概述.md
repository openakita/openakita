**③ ClickHouse → StarRocks**

| 维度 | ClickHouse | StarRocks | 影响 |
|------|-----------|-----------|------|
| 协议 | HTTP/native | **MySQL 协议** | Go driver 从 clickhouse-go → go-sql-driver/mysql，学习成本↓ |
| SQL 方言 | ClickHouse SQL | MySQL 兼容 SQL | 现有团队 MySQL 经验可直接复用，ETL 脚本几乎无改动 |
| 运维 | 自建集群复杂 | 存算分离、FE/BE 架构 | 部署简化，但需学习 StarRocks 特有运维（FE 元数据管理） |
| 生态 | Altinity Cloud | StarRocks Cloud（阿里云/私有化） | AWS 区域需自建，暂无原生 managed service |
| 成本 | 中 | 中（略低） | 同等规模硬件，StarRocks 资源利用率更高 |
| 多租户 | tenant_id 列分片 | 同左（推荐 Primary Key 表） | 策略不变，表引擎从 MergeTree → Primary Key |
| 风险 | 社区成熟 | 社区较新，文档偏少 | 安排数据分析师前期跑 benchmark 验证 |

**净影响评估**：
- ✅ 开发效率提升（MySQL 协议降低接入门槛）
- ✅ 数据分析师可独立写 SQL 验证，不依赖后端
- ⚠️ 运维知识需额外学习（建议 Week 1 安排 2 天 spike）
- ⚠️ AWS 自建部署，无 managed service 兜底

## 2. 技术栈

### 2.1 后端

| 组件 | 选型 | 理由 |
|------|------|------|
| 主语言 | Go 1.22+ | 高并发、低资源消耗、部署简单（单二进制） |
| 服务间通信 | gRPC + Protobuf | 强类型、高性能、代码生成友好 |
| HTTP 网关 | gRPC-Gateway / Connect | 对外暴露 REST，内部走 gRPC |
| 认证 | JWT + OAuth2 | 中小企业常用 IdP 对接 |
| 任务调度 | 异步队列（NATS / Redis Streams） | 财务报表生成等异步任务 |

### 2.2 前端

| 组件 | 选型 | 理由 |
|------|------|------|
| 框架 | Next.js 14+ (App Router) | SSR/SSG、RSC 支持、生态成熟 |
| UI 库 | shadcn/ui + Tailwind CSS | 快速搭建、可定制 |
| 图表 | Apache ECharts / Recharts | 财务数据可视化核心能力 |
| 状态管理 | Zustand / TanStack Query | 轻量、服务端缓存友好 |

### 2.3 数据层

| 数据库 | 用途 | 关键特性 |
|--------|------|----------|
| PostgreSQL 16 | 业务库（用户、权限、配置、订单） | ACID、成熟生态 |
| **StarRocks** | 分析引擎（财务指标、报表、趋势） | 列式存储、MySQL 协议兼容、亚秒级聚合 |
| Redis 7 | 缓存 + 会话 + 限流 | 高速读写 |
| S3 | 文件存储（导入导出、报表 PDF） | 成本低、与 AWS 集成 |

## 3. 服务拆分（gRPC 微服务）

| 服务 | 职责 | 对应 DB |
|------|------|---------|
| `user-svc` | 用户注册/登录/权限/RBAC | PostgreSQL |
| `org-svc` | 企业/组织/多租户管理 | PostgreSQL |
| `data-svc` | 数据导入/清洗/映射（银行流水、ERP 对接） | PostgreSQL + S3 |
| `finance-svc` | 财务指标计算/规则引擎 | PostgreSQL → **StarRocks** |
| `analytics-svc` | 分析查询/报表生成/趋势预测 | **StarRocks** |
| `report-svc` | 报表模板/导出(PDF/Excel)/定时推送 | S3 + PostgreSQL |
| `ai-svc` | 智能洞察/NL Query/异常检测 | **StarRocks** + LLM API |
| `gateway` | HTTP→gRPC 转换 + 认证 + 限流 | Redis |

## 7. 关键技术决策

### 7.1 StarRocks vs 其他 OLAP

选择 StarRocks 的核心理由：
- **MySQL 协议兼容**: 团队现有 MySQL 经验直接复用，Go driver 用标准 `go-sql-driver/mysql`
- **查询速度**: 列式存储 + 向量化引擎，亿级聚合 < 1秒
- **成本**: 自建集群 vs Redshift/Snowflake SaaS 费用，中小企业预算友好
- **Primary Key 表**: 支持实时 Upsert，适合财务数据的修正场景
- **存算分离**: FE（元数据）+ BE（数据）架构，弹性扩展友好

### 7.2 Go 微服务 vs 单体

MVP 阶段采用 **模块化单体 + gRPC 接口定义**，服务间先以 Go module 隔离，后续按需拆分独立部署。避免过早微服务化带来的运维负担。

### 7.3 多租户策略

采用 **Schema-per-tenant**（PostgreSQL）+ **tenant_id 列**（StarRocks）：
- PG: 每个企业独立 schema，数据隔离强
- StarRocks: 分布式表按 tenant_id 分桶，查询自动路由
